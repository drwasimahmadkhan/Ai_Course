Constraint Satisfaction Problems (CSPs)

CSPs are a special class of problems with structured representation that enables powerful solving techniques.

CSP Definition

A CSP consists of: Variables (X), Domains (D), and Constraints (C)

Three Core Components

Variables
Set of variables {X₁, X₂, ..., Xₙ} to be assigned values
Domains
Each variable Xᵢ has a domain Dᵢ of possible values
Constraints
Restrictions on allowed combinations of values (e.g., X₁ ≠ X₂)

Types of Domains

🔢

Finite Discrete

Fixed set of values. Example: Map coloring with {R, G, B}.

♾️

Infinite Discrete

Unbounded integers. Example: Job scheduling with time variables.

📏

Continuous

Real-valued domains. Example: Robot joint angles.

Classic CSP Examples

  • Map Coloring: Color regions so no adjacent regions share colors
  • N-Queens: Place N queens on board so none attack each other
  • Sudoku: Fill grid with digits 1-9, each appearing once per row/column/box
  • Scheduling: Assign time slots to exams respecting student conflicts

Key Advantage Structured representation allows constraint propagation — much faster than generic search.

Constraint Propagation & Inference

Before or during search, we can eliminate impossible values using local consistency checks.

Local Consistency Types

Node Consistency

For each variable X, remove values that violate unary constraints (constraints involving only X).

Arc Consistency (AC)

For each pair of variables (Xᵢ, Xⱼ): if value v in Dᵢ has NO compatible value in Dⱼ, remove v from Dᵢ.

AC-3 Algorithm

Pseudocode

1. Initialize queue with all arcs (Xᵢ, Xⱼ)
2. While queue not empty:
   • Remove arc (Xᵢ, Xⱼ)
   • Remove values from Dᵢ incompatible with Xⱼ
   • If Dᵢ changed, add all arcs (Xₖ, Xᵢ) to queue
3. Return reduced domains

Higher-Order Consistency

Path Consistency
For any consistent 2-variable assignment, can extend to 3rd variable
k-Consistency
For any consistent (k-1)-variable assignment, can extend to kth variable
Strong k-Consistency
Strong k-consistency + all lower-order consistencies satisfied

Computational Trade-off Higher consistency is more expensive but reduces search needed.

Backtracking Search for CSPs

CSP backtracking is similar to generic search but uses domain-specific optimization techniques.

Backtracking Algorithm

Assign variables one by one; backtrack when constraint violation detected. Much faster than generic search on CSPs.

Critical Optimizations

1. Variable Ordering Heuristics

MRV (Minimum Remaining Values)
Choose variable with smallest domain. Most constrained variable first → fail fast.
Degree Heuristic
Choose variable involved in most constraints with unassigned variables.

2. Value Ordering Heuristics

Least Constraining Value (LCV)
Try values that eliminate fewest options for neighboring variables. Maximizes chances of finding solution.

3. Inference During Search

Forward Checking
When assigning X, delete values from neighbors' domains that conflict.
MAC (Maintaining Arc Consistency)
Run AC-3 after each assignment to detect inconsistency early.

Improvement MRV + Forward Checking can solve problems 1000x faster than basic backtracking!

Local Search for CSPs

When we only need a solution (any valid assignment), not a path, local search is very effective.

Key Insight

Start with a complete (possibly invalid) assignment, then improve by fixing violated constraints.

Min-Conflicts Heuristic

Algorithm

1. Start with random complete assignment
2. While assignment has conflicts:
   • Pick random conflicted variable
   • Assign value that minimizes total conflicts
3. Return solution when all constraints satisfied

Empirical Results

N-Queens Problem: Min-conflicts solves 1,000,000-queens in ~50 assignment changes!

Advantages

• Very fast for many CSPs
• Scales to large problems
• Natural for real-time applications
• Memory efficient

Disadvantages

• Not complete (can get stuck)
• Not guaranteed to find solution
• Depends on initial assignment

Application Scheduling, timetabling, and real-time constraint solving.

Games and Adversarial Search (Chapter 6)

In games, we have an opponent trying to minimize our success. Search must plan against adversarial play.

Game Definition

Formal Game Specification

• Initial state: Starting position
• To-Move(s): Whose turn
• Actions(s): Legal moves
• Result(s, a): State resulting from action
• Terminal-Test(s): Game over?
• Utility(s, p): Final payoff for player p

Two-Player Zero-Sum Games

Perfect-information, deterministic games where one player's gain = other's loss (e.g., chess, checkers, tic-tac-toe).

♟️

Perfect Information

Both players see entire game state at all times.

🎯

Deterministic

No chance/randomness in outcomes (unlike backgammon).

⚖️

Zero-Sum

Payoffs sum to zero: U(s, MAX) = -U(s, MIN)

Minimax Algorithm

Find optimal play assuming opponent also plays optimally using backward induction from terminal states.

Core Idea

MAX maximizes score; MIN minimizes score. Minimax value = best outcome assuming optimal play.

Algorithm Steps

Minimax Pseudocode

Minimax(s):
  if Terminal-Test(s): return Utility(s)
  if Max-To-Move(s):
    return max(Minimax(Result(s, a)) for a in Actions(s))
  else:
    return min(Minimax(Result(s, a)) for a in Actions(s))

Complexity Analysis

Time Complexity
O(bm) where b = branching factor, m = depth
Space Complexity
O(bm) — storing one path at a time
Practical Limitation
Chess: b ≈ 35, m ≈ 100 → 35100 is enormous!

Example Tic-tac-toe search tree has ~9! ≈ 362K terminal states—minimax is feasible.

Alpha-Beta Pruning

Dramatically reduce search by eliminating branches that cannot affect final decision.

Key Principle

Track bounds: α = best MAX can guarantee, β = best MIN can guarantee. Prune when value exceeds bounds.

How It Works

Pseudocode Snippet

Max-Value(s, α, β):
  if Terminal-Test(s): return Utility(s)
  v = -∞
  for each a in Actions(s):
    v = max(v, Min-Value(Result(s,a), α, β))
    if v ≥ β: return v // β-cutoff
    α = max(α, v)
  return v

Performance Impact

With Random Move Ordering

Eliminates ~90% of nodes.
Effective branching factor: √b

With Perfect Ordering

Reach depth 2d instead of d in same time!

Move Ordering Matters Best moves first → more pruning. Use killer moves, transposition tables, and iterative deepening.

Evaluation Functions & Enhancements

Practical game AI cuts off search at non-terminal states and uses static evaluation functions.

Cut-Off Depth Search

Strategy

Search to fixed depth d, use evaluation function to estimate utility of non-terminal positions. Much faster!

Evaluation Function Design

Linear Combination
f(s) = w₁f₁(s) + w₂f₂(s) + ... + wₙfₙ(s) where fᵢ are features, wᵢ are weights.
Chess Example
Material: piece values (pawn=1, knight=3, etc.). Piece position, king safety, control of center.

Advanced Search Enhancements

🌳

Transposition Tables

Cache position evaluations to avoid re-computing identical states reached via different move orders.

⚡

Iterative Deepening

Deepen search gradually; refine move ordering between iterations for better pruning.

🔍

Quiescence Search

Extend search in volatile positions (e.g., during tactical exchanges) to avoid horizon effect.

Monte Carlo Tree Search (MCTS)

Alternative to minimax: balance exploration and exploitation using random playouts.

MCTS Phases

Selection
Navigate tree using UCB1 policy until reaching leaf node.
Expansion
Create new child node for unexplored move.
Simulation
Play out random (or guided) moves to terminal state.
Back-propagation
Update statistics (wins, visits) up the tree.

UCB1 Formula (Upper Confidence Bounds)

Node Selection

UCB1(n) = U(n)/N(n) + C × √(log N(parent)/N(n))
• Exploitation: U(n)/N(n) = win rate
• Exploration: √(log N(parent)/N(n)) = inverse visit frequency
• C: Exploration constant (typically 1/√2)

MCTS Advantages

  • No evaluation function needed — uses only game rules
  • Naturally handles high-branching games (Go: ~250 moves/position)
  • Anytime algorithm — returns best move so far if interrupted
  • Parallelizable — multiple threads can search simultaneously

Historic Win AlphaGo used MCTS + deep learning to beat world champion at Go!

Stochastic Games and Uncertainty

Real games often include chance nodes (dice rolls, card shuffles, etc.).

Expectiminmax Value

At chance nodes, average utility over all outcomes weighted by probability.

Game Tree with Chance

Expectiminmax Algorithm

Expectiminmax(s):
  if Terminal-Test(s): return Utility(s)
  if Max-To-Move(s):
    return max(...)
  if Min-To-Move(s):
    return min(...)
  if Chance-Node(s):
    return Σ P(outcome) × Expectiminmax(Result(s, outcome))

Backgammon Example

Dice rolls create chance nodes. Expected value = weighted average of all possible dice outcomes. More branches to evaluate, but same recursive structure as minimax.

Computational Impact Chance nodes multiply branching factor, making deeper search infeasible for most games.
